# Gotchas

- HF Trainer monkey-patch in `train/trainer.py` — fragile; breaks if `Trainer.__init__` changes.
- Hard-coded library paths in `orchestrate/env.py` — cluster-specific.
- No dataset validation in `build_phase1_dataset()` — empty QA pairs silent-fail.
- Tulu pipeline caveat: midtraining+Tulu results may not generalize to production post-training.
- **`+gpu_id=N` Hydra override required for multi-GPU parallel training launches.** `train/sft.py` sets `os.environ["CUDA_VISIBLE_DEVICES"] = str(cfg.gpu_id)`, clobbering any env `CUDA_VISIBLE_DEVICES` (default `0` → all parallel jobs on GPU 0 → OOM). Pass `+gpu_id=N` per process (the `+` is required — `gpu_id` isn't in the default schema).
- **The in-process `CUDA_VISIBLE_DEVICES` clobber is silently defeated by any import-time cuInit — parallel per-GPU launches must ALSO pin it in the LAUNCHER environment.** The driver freezes its device list at the FIRST cuInit in the process; if any import in the dispatcher chain initializes CUDA (`import peft` is a known offender — incident #545, 2026-06-11) before `train_lora`'s `os.environ["CUDA_VISIBLE_DEVICES"] = str(cfg.gpu_id)` (`train/sft.py`), the late clobber is a driver-level no-op and every process's `cuda:0` resolves to physical GPU 0 — parallel cells co-locate and OOM. Neither `torch.cuda.is_initialized()` returning `False` nor `torch.cuda.device_count()` returning the post-clobber count proves the pin took (both read the clobbered env, not the frozen driver state). RULE: export `CUDA_VISIBLE_DEVICES=<gpu>` per cell in the launcher environment before exec, keeping the matching `--gpu <gpu>` / `+gpu_id=<gpu>` arg so the in-process clobber rewrites the same value — never rely on the in-process clobber alone. Diagnostic signature: all parallel cells OOM on ONE physical GPU whose OOM message lists every sibling PID. (Incident #543 ratio-1pct-arm relaunch-2, 2026-06-10: 3 phase-1 cells with distinct `--gpu` args co-located on GPU 0 and OOM'd; a clean process honored the same late CVD set, the script's import chain did not.) Experimenter-side enforcement: `.claude/agents/experimenter.md` Before Running step 10; regression smoke: `tests/test_cvd_wave_assignment_smoke.py`; lane-independent mechanical enforcement: `scripts/workflow_lint.py --check-dispatcher-cvd-pin` (bundled into the no-flags default run) FAILs any backgrounded `--gpu-id`/`+gpu_id=` python launch in `scripts/*.sh` without a `CUDA_VISIBLE_DEVICES=` prefix on the same command — waive legitimate shapes with `# CVD_PIN_EXEMPT: <reason>`.
- **RunPod MooseFS per-pod disk quota (~130 GB), separate from share-level free space.** `df -h /workspace` shows the share size (TB free) but each pod has a ~130 GB writable quota; writes past it fail with `OSError errno=122 (EDQUOT)` (`shutil.disk_usage` misses this — preflight uses a `posix_fallocate` probe instead). Symptoms: log appends fail with "Disk quota exceeded", WandB inline uploads emit Errno 122, checkpoint loads die silently. Mitigations: `EPM_SKIP_INLINE_CHECKPOINT_UPLOAD=1` for sweeps; sequentialize multi-condition sweeps; delete `coupling_merged/` after each phase; provision a bigger pod for 6+ Qwen-7B checkpoints.
- **MooseFS/FUSE D-state "zombie" processes on write-heavy runs.** Trainer + WandB writes hammering the FUSE mount can wedge processes in uninterruptible-sleep (`D` state) that even `kill -9` cannot reap; the GPUs sit idle while the run looks alive, and the only drain is a pod `stop` + `resume`. Standing mitigation for write-heavy runs: route trainer output AND WandB tempfiles OFF the FUSE mount — `EPM_OUTPUT_ROOT=/tmp/issue_<N>_results` + `WANDB_MODE=offline` (sync WandB afterward) — so the hot write path is local disk, not MooseFS. (Incident #505, 2026-06-08: pod-505 needed repeated stop+resume to drain D-state zombies until output was rerouted off FUSE.)
- **vLLM in-process teardown does NOT reap worker subprocesses.** When the SAME process loads vLLM then a non-vLLM framework (HF Transformers, sentence-transformers), the canonical cleanup (`del llm` + `destroy_model_parallel()` + `destroy_distributed_environment()` + `gc.collect()` + `empty_cache()`) is NOT enough — vLLM TP/PP worker subprocesses survive and re-grab the freed GPU memory the moment the next framework loads weights (looks like an HF-Transformers OOM). Add: (a) `psutil.Process().children(recursive=True)` → `.terminate()` then `.kill()` survivors; (b) `nvidia-smi --query-compute-apps=pid` → FAIL LOUD if any python PID still holds the GPU. Escape hatch: if switching frameworks >twice, subprocess-isolate each phase (JSON IPC on disk).
- **Crashed vLLM parents leave orphaned `VLLM::EngineCore` workers that OOM the RELAUNCH — and `pgrep -f <script-name>` cannot see them.** The crash-orphan variant of the teardown trap above: when a vLLM workload crashes (or is killed), its engine-worker subprocesses can outlive the dead parent and silently keep holding ~50 GB on EVERY GPU, so the next launch dies at engine init with `Free memory on device (...) is less than desired GPU memory utilization`. The workers' cmdline is just `VLLM::EngineCore` (no script name, no python path), so the natural pre-relaunch liveness probe `pgrep -f <script-name>` reads clean while the GPUs are full. RULE: before ANY vLLM relaunch, probe `nvidia-smi --query-compute-apps=pid,used_memory --format=csv` AND `pgrep -af EngineCore`, kill any leftovers (`kill`, then `kill -9` survivors), and confirm GPU memory is ~0 before launching the new driver. Experimenter-side enforcement: `.claude/agents/experimenter.md` Pre-Launch step 9. (Incident #601, 2026-06-11: 4 orphaned EngineCore workers from a phase0 crash held ~50 GB/GPU and OOMed the hot-fix relaunch until they were killed.)
- **Before stopping/terminating a pod you believe is idle/done, verify with the live RunPod API + `nvidia-smi` GPU utilization — never infer "idle" from an SSH-unreachable pod or an absent `epm:run-launched` marker.** Diagnostic and manual runs emit no `run-launched` marker, and an SSH refusal usually means the cached host/port drifted (a `pod.py resume` changes them), NOT that the pod is free — both signals falsely read "nothing running." Run `pod.py config --sync` to repair any address drift, then confirm ~0% GPU utilization across ALL devices via `nvidia-smi` before any `pod.py stop` / `terminate`. (Incident #488, 2026-06-08: a live diagnostic training run was killed because SSH-refused + no `run-launched` marker were misread as "idle.")
- **SSH MCP `ssh_execute` caps commands at ~30 s by default — and `timeout` must be a NUMBER (ms).** A pod command wrapped in `timeout 240 sh -c ...` still dies at the MCP layer with `Command timeout after 30000ms` unless you pass the tool's own `timeout` param ≥ the inner timeout; passing it as a string raises `Expected number, received str`. For anything that can exceed ~30 s, prefer a bg-Bash `ssh` call. (4 hits on 2026-06-09, issue #521/#522 sessions.)
- **Launching a long pod job over SSH: redirect stdin or the ssh call hangs.** `ssh pod-<N> 'nohup ... > log 2>&1 &'` keeps the SSH session open on the backgrounded child until an outer `timeout` kills it (exit 124 = the ssh timeout, not the job). Use the full idiom `nohup ... > log 2>&1 < /dev/null &` then exit immediately, and verify the job separately via the PID file / log mtime. (Incident #532 launch, 2026-06-09.)
- **`python-dotenv` no-arg `load_dotenv()`/`find_dotenv()`: stdin heredocs CRASH; `-c` one-liners silently load nothing off-repo.** From a stdin-fed heredoc (`uv run python - <<PY`), no-arg `find_dotenv()` walks the call stack (`frame.f_back`) and dies on `assert frame.f_back is not None`. Under `python -c` it does NOT crash (verified against the pinned python-dotenv 1.2.2, 2026-06-12: `__main__` has no `__file__` under `-c`, so `_is_interactive()` short-circuits to a cwd-walk and the frame walk is never reached) — the `-c` failure mode is instead SILENT: run from a cwd not under a dir containing `.env`, `load_dotenv()` returns False, no keys load, and the missing-API-key crash fires later and far from the cause. Both shapes: pass the explicit path `load_dotenv("/path/to/.env")`, use the stdin-safe project wrapper `research_workflow.orchestrate.env.load_dotenv()`, or `set -a && source .env && set +a` in the enclosing shell. (3 hits on 2026-06-09 — heredoc-shape; reintroduced on #612, 2026-06-12.) The heredoc half is enforced mechanically for `scripts/*.sh` by `scripts/workflow_lint.py --check-heredoc-dotenv` (bundled into the no-flags default run); the `-c` half is DELIBERATELY not lint-covered (a no-arg call from the repo root works — cwd-walk finds `.env` — so a hard FAIL would flag legitimate shapes; extension considered + rejected 2026-06-12). Prefer the explicit path in `-c` one-liners anyway.
- **Pod/VM driver scripts that default `REPO_ROOT` to the RunPod path die on the GCP lane unless the override is threaded at dispatch.** The GCE startup script clones the repo to `$WORKLOAD_ROOT` (`/workspace/wf-issue-<N>`) and runs the workload command from there, so a driver with `REPO_ROOT="${REPO_ROOT:-/workspace/your-project}"` fails at its first `cd` under `set -e` and the EXIT trap powers the VM off. RULE: compose gcp/auto dispatches as `--workload-cmd 'REPO_ROOT="$WORKLOAD_ROOT" bash scripts/<driver>.sh'` (incident #599, 2026-06-11: ~8 min GPU burn, guest-attribute `phase=failed`).
- **vLLM `prompt_logprobs` + large `max_model_len` OOMs on float32 logits materialization.** With `prompt_logprobs` enabled, vLLM materializes float32 logits for every token-slot in a scheduler step; at `max_model_len=32768` with uncapped `max_num_batched_tokens` that is a ~19 GiB single allocation inside `logits.log_softmax` — `gpu_memory_utilization` tweaks alone do NOT fix it. RULE: any prompt-logprobs eval sets `max_num_batched_tokens≈8192` AND `gpu_memory_utilization<=0.75` on the `LLM()` init. (Incident #501, 2026-06-06: two CUDA OOMs, ~1h wall lost; fix commit cb4e4119f ran clean.)
- **`orchestrate.preflight --json` emits PRETTY-PRINTED multi-line JSON — parse the WHOLE stdout, never `splitlines()[-1]`.** The CLI prints `json.dumps(..., indent=2)`, so the final stdout line is a bare `}`; a dispatcher gate doing `json.loads(proc.stdout.splitlines()[-1])` crashes even when preflight PASSes. Parse with `json.loads(proc.stdout)` (the CLI prints exactly one JSON object and nothing else on stdout). RULE: dispatchers that gate on preflight ALSO expose a `--skip-preflight` flag so a falsely-failing gate can be bypassed in-turn after manual preflight verification, instead of costing a relaunch cycle. (Incident #602, 2026-06-11: pod launch died <1 min in on the bare `}` despite a PASSing preflight; the experimenter verified manually and relaunched with `--skip-preflight`.)
- **HF mirror ≠ local-verified copy — sha256-pin every reused-artifact prefetch.** A worker-side dispatcher that prefetches REUSED artifacts from a shared HF inherit path (e.g. a parent task's `issue<M>_<slug>/...` on the data repo) can pull a silently different generation than the local copy the plan and reviewers verified: local `data/` outputs are untracked (absent from every clone), and the standard pre-flight pair — verify content LOCALLY + verify the artifact RESOLVES on HF (`list_repo_files` / HEAD) — never content-compares the two. The divergence surfaces as a KeyError (or a wrong persona universe) deep in the consumer, after a full provision cycle is already spent. RULE: snapshot the verified local inputs to an issue-OWNED HF path (`issue<N>_<slug>/inputs/`) instead of consuming the parent's shared mirror; pin every prefetch/autofetch against an `EXPECTED_SHA256` table asserted at the trust boundary — covering files already present on the worker, not just fresh downloads — and add a consume-time coverage assert where the artifact is read. (Incident #600, 2026-06-11: stale `dac5749`-generation HF mirrors of #472's `R_train.json` + `centroids_L10.pt` crashed the GCP smoke run; fix pattern: the `EXPECTED_SHA256` pins in #600's `dispatch.py`.)
- **GCE metadata script runner kills GCP-lane workloads on giant progress-bar lines — and the VM zombies at RUNNING.** `google_metadata_script_runner` reads startup-script stdout line-by-line with a bounded Go `bufio.Scanner`; a long NEWLINE-FREE line — vLLM/tqdm `\r`-carriage-return progress bars ("Processed prompts: ...") are the canonical case — overflows it (syslog signature: `error while communicating with "startup-script" script: bufio.Scanner: token too long` then `Script "startup-script" failed with error: signal: broken pipe`) and the workload dies on SIGPIPE. The EXIT trap RUNS but `$?` inside it reads 0 (status of the last completed command — measured on bash 5.1.16, #607 fact-check), so the `[ "$rc" -ne 0 ]` guard no-ops and the `eps/phase` guest attribute never flips to `failed` and the VM stays RUNNING (and billing) with the GCP poll reading a healthy "workload" phase forever — indistinguishable from a healthy long run. Detect: GPUs at 0% + no workload python processes + `sudo grep "token too long" /var/log/syslog`; disk state survives (only the parent shell was killed). Recover: SSH-relaunch the REMAINING phases on the same VM under `nohup` with stdout/stderr redirected to a FILE (bypasses the metadata-runner pipe entirely), then on success replicate the startup script's success tail — write the completion sentinel at the handle's `sentinel_path` and publish `eps/phase=done` via the guest-attribute curl — so the existing poll/finalize path proceeds unchanged; keep the rc!=0 → `_eps_phase failed; shutdown -h now` trap for billing bounds. Prevent: never stream raw workload output through startup-script stdout — redirect the workload block to a log file (the `log_path` the handle already names) and/or set `TQDM_DISABLE=1` / vLLM `disable_tqdm` on non-tty. The renderer-side fix (`src/research_workflow/backends/gcp.py` `render_startup_script`) is tracked as a separate infra task — this entry documents the trap. (Incident #491 attempt 2, 2026-06-11: killed mid-free_gen after 22/29 cells; recovered via SSH nohup relaunch on the live VM.)
- **Lazy imports inside smoke-skipped branches (`--dry-run` / `--skip-upload` / GPU-only paths) are unverified code — the ImportError fires on the pod, AFTER the expensive phases.** Local smokes legitimately skip upload/GPU branches, so a deferred `from ... import` there executes for the first time in production; an assumed-but-absent symbol (canonical case: porting code that calls a private file-local helper — `_retry_transient` was defined locally in the i528 scripts, never in `orchestrate/hub.py`) crashes the workload only after training/judging are already spent. RULE: hoist cheap cross-script helper imports to module top (absence then crashes at process start); gate relaunches on an AST-based `--verify-imports` smoke that walks every in-scope file and EXECUTES every deferred import (hand-maintained symbol lists just re-create the drift — generate the list from the AST); and when porting code that calls a private `_underscore` helper, grep the SOURCE branch for where the symbol is actually DEFINED — file-local helpers don't travel with the import path you assume. Worked example: `scripts/issue_606/i606_dispatch.py --verify-imports` (commit `c39627b30` — CPU-only, no GPU/API; 49 deferred symbols across 7 files, exit-0 gate). (Incident #606, 2026-06-11: GCP workload crashed at p5_upload ~18 min in, after training + stage-A judging, on `ImportError: cannot import name '_retry_transient'` — every local smoke had skipped the upload branch.)
- **Smoke-root rebinding (`--smoke` → `*_smoke` GEN/OUT/EVAL roots) orphans READ-ONLY PARENT INPUTS.** Smoke/real root isolation is for GENERATED artifacts; parent inputs (clouds / caches / pools fetched from a prior task) are inputs, not outputs. If the fetch step stages them under the real root only — or a fetch early-return skips the staging under `--smoke` — while a downstream step still consumes them, a fresh-pod smoke crashes at the first consuming step on a path only the real run staged (signature: an assertion on the first parent item AFTER all earlier smoke steps passed). RULE: give parent inputs a canonical NON-rebinding location (e.g. `CLOUDS_PARENT = .../eval_results/issue_<N>/clouds_parent`) staged BEFORE any smoke early-return, so identical fetch+read lines execute in both modes; never fix with a smoke-mode skip (removes coverage) or a read-side-only fallback (a fresh pod running smoke-first still has nothing staged); when writing any `--smoke` rebinding dispatcher, audit every downstream read for parent-input paths and pin them to non-rebinding constants. Sibling trap from the same family: the deliberate root isolation means a completed REAL early phase does NOT populate the `*_smoke` roots — a mid-chain smoke stage (`--phase train --smoke`) hard-requires its own early-phase `--smoke` prestage on the same pod (signature: ~6 s death at the build step on a missing smoke input file); stat-check the smoke GEN root before launching any mid-chain smoke stage. (Incident #542, 2026-06-12: `--phase p0prime --smoke` died at the closeness step with `AssertionError: cloud missing for sp_swe` — the real run had staged `clouds_parent/` under the real root; a second launch the same day burned on the missing-prestage sibling.)
- **vLLM `max_model_len` must track every `max_new_tokens` deviation on an inherited eval rig.** On-policy slot reads re-enter the generated response as a PROMPT (prompt + R + marker), and vLLM validates prompt length against `max_model_len` — so the moment a generation hits the `max_new_tokens` cap, the follow-on read exceeds the cap by construction and the eval worker crashes (`ValueError: ... decoder prompt ... longer than the maximum model length`). Bumping `max_new_tokens` (e.g. for the ≥2× longest-completion rule) while inheriting a parent module that pins `max_model_len` / `DEFAULT_MAX_MODEL_LEN = 2048` re-creates the crash, and a CPU smoke structurally cannot reach a cap-length generation — only the pod run surfaces it. RULE: when a plan names a max-token deviation on an inherited rig, grep the parent module for `max_model_len` / `DEFAULT_MAX_MODEL_LEN` and pass an explicit value ≥ longest prompt + `max_new_tokens` + slot overhead at the call site (≈2× `max_new_tokens` is the working margin), leaving the shared parent default untouched. Recurring: #505 round 9 (2026-06-08, `eval_trajectory_505.py` overrides at its call site only) and #601 Phase 0 (2026-06-11, hot-fix `23f5812ad`) hit the identical overflow against #472's `DEFAULT_MAX_MODEL_LEN = 2048` in `contrastive_neg_geometry_472/eval_trajectory.py`.
- **Stale `VIRTUAL_ENV` from another project (e.g. `~/other-project/.venv`) makes every `uv run` print a mismatch warning — it is noise, not a fault.** Sessions inherit `VIRTUAL_ENV` from the spawning shell/daemon environment; when it points at a different project's venv, `uv run` prints `warning: VIRTUAL_ENV=... does not match the project environment path .venv and will be ignored` on EVERY invocation (381 hits across one day's session transcripts, 2026-06-11). uv resolves the correct project venv regardless — do NOT chase the warning, repair the venv, or blame it for an adjacent crash (one session's same-output IndexError was an unrelated script bug). Only real effect: a script that READS `$VIRTUAL_ENV` directly sees the wrong path — run `env -u VIRTUAL_ENV uv run ...` where that matters.
